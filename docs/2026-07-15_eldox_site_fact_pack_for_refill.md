## 5. Чего пока нельзя писать без подтверждения

Нельзя без отдельного подтверждения:

- "официальный партнер Saby / СБИС";
- "сертифицированный партнер";
- "лучшие цены";
- "самый быстрый запуск";
- "гарантируем результат";
- "работаем по всей России", если нет подтвержденной географии;
- "N клиентов", "N внедрений", "N лет опыта", если цифры не даны;
- реальные отзывы, логотипы клиентов, названия компаний без разрешения;
- точные цены Элдокс по регионам и лицензиям, пока не подтверждена модель.

Можно осторожно:

- "помогаем подобрать и настроить Saby под задачу";
- "разбираем регион, сценарий, пользователей и состав работ";
- "поддерживаем запуск и сопровождение после подключения";
- "Saby, ранее СБИС" при первом упоминании.

## 6. Типовые кейсы и отзывы: правила генерации

Owner разрешил кейсы и отзывы пока "придумывать". Internal Control принимает это только при честной маркировке:

- не "отзыв клиента", а "типовая ситуация";
- не "компания X получила результат", а "как может выглядеть сценарий";
- не выдумывать имена, должности и логотипы;
- можно использовать обезличенные форматы: "торговая компания", "бухгалтерская фирма", "сервисная компания", "магазин с маркировкой".

### 6.1 Типовые сценарии для сайта

1. Торговая компания переходит на ЭДО.
   - Ситуация: контрагенты уже отправляют документы электронно, часть документов теряется в почте.
   - Работы: подобрать сценарий Saby Docs, настроить роли, маршруты, подписи, порядок обмена.
   - Результат: документы проходят по понятному маршруту, руководитель видит статус, бухгалтерия не собирает оригиналы вручную.

2. Компания сдает отчетность в ручном режиме.
   - Ситуация: сроки держатся на одном сотруднике, требования и сверки отслеживаются вручную.
   - Работы: настроить Saby Report, календарь, уведомления, проверки, доступы.
   - Результат: команда заранее видит сроки и ошибки, требования не пропадают.

3. Saby уже есть, но команда его обходит.
   - Ситуация: лицензии подключены, но сотрудники продолжают работать через Excel, почту и мессенджеры.
   - Работы: аудит сценария, роли, маршруты, обучение, инструкции.
   - Результат: сервис становится частью процесса, а не отдельной вкладкой.

4. Сайт есть, заявки теряются.
   - Ситуация: форма отправляет письма, менеджеры пересылают обращения вручную.
   - Работы: уточнить оффер, форму, уведомления, источник, CRM/таблицу, ответственного.
   - Результат: заявка фиксируется и получает следующий шаг.

5. CRM есть, но руководитель не видит картину.
   - Ситуация: статусы ведутся нерегулярно, источники и причины потерь не видны.
   - Работы: пересобрать воронку, статусы, обязательные поля, уведомления, отчеты.
   - Результат: у каждой заявки есть владелец и следующий шаг.

## 7. Фактура для визуального спринта

### 7.1 Обязательные визуальные активы

1. Главный hero:
   - смысл: документы, заявки, CRM и цифровой маршрут в одном рабочем процессе;
   - стиль: темный акцент допустим, но без абстрактных "техно-кубов" без смысла;
   - должен поддерживать текст о Saby как точке входа и цифровизации как горизонте.

2. Saby / ЭДО:
   - документы, подпись, маршрут согласования, контрагент, статус;
   - можно использовать инфографику, не реальные скриншоты без разрешения.

3. Отчетность:
   - календарь сроков, проверка, уведомления, госорганы.

4. Кадры:
   - сотрудник, личный кабинет, кадровый документ, подпись.

5. Сайты и заявки:
   - страница -> форма -> уведомление -> CRM/таблица.

6. CRM:
   - воронка, карточка заявки, статус, ответственный, следующий шаг.

7. База знаний:
   - уникальные обложки по рубрикам: Saby, сайты, CRM, автоматизация.

### 7.2 Что не использовать

- одинаковый `hero.jpg` на всех статьях;
- случайные стоковые офисы;
- абстрактный код/схемы микросхем;
- неоновые фоны для плотных информационных блоков;
- изображения, которые создают ощущение "ИИ-сайта", а не B2B-партнера.
