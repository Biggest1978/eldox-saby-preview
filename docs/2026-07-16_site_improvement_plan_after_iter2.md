# Детализированный план улучшений сайта после второй тройственной приемки

Дата: 2026-07-16  
Основание: вторая тройственная приемка сайта Элдокс, вердикт — текущая версия не проходит внутренний контроль.  
Цель итерации: довести сайт до состояния, где каждая страница и каждый ключевой блок выглядят профессионально, понятны заказчику и конечному пользователю, усиливают доверие и ведут к заявке.

## Принцип следующей итерации

Исправляем не “точечные слова”, а пользовательское восприятие:

1. Визуальная логика должна быть объяснима без комментария разработчика.
2. Текст должен звучать как клиентская польза, а не как внутреннее ТЗ.
3. Доказательства должны показывать проверяемый результат, а не обещание “мы все настроим”.
4. Любой выделенный элемент должен иметь функцию: активный фильтр, главный сценарий, рекомендуемая точка входа, результат, предупреждение или CTA.
5. После правок проводится скриншотная приемка по всем страницам, а не только поиск запрещенных слов.

## Приоритеты работ

### P0 — блокеры приемки

Эти правки обязательны до следующего показа результата как “почти готового”.

1. Убрать служебный язык из публичных текстов.
2. Исправить необоснованное выделение карточек.
3. Пересобрать доказательные блоки.
4. Развести визуальные паттерны похожих блоков.
5. Обновить витрину публикаций: разные изображения, разные смысловые крючки, не шаблонная карточная лента.

### P1 — усиление маркетинга и SEO

1. Расширить тексты проблемных блоков до уровня “клиент узнает себя”.
2. Добавить больше конкретики по сценариям: документы, заявки, CRM, сайт, автоматизация.
3. Привязать каждый CTA к текущему состоянию пользователя.
4. Усилить внутреннюю перелинковку между главной, услугами и статьями.

### P2 — визуальная полировка

1. Подготовить отдельные обложки для статей.
2. Добавить больше визуальных различий между секциями: темный proof, светлая витрина, редакционный блок, процессная схема, карточки услуг.
3. Проверить desktop и mobile на ритм, плотность, пустоты и повторяемость.

## Новая драматургия главной

1. **Hero:** Элдокс помогает запустить Saby и цифровые решения под реальные процессы бизнеса.
2. **Боли:** клиент узнает свою ситуацию: документы, отчетность, заявки, CRM/Excel.
3. **Saby как точка входа:** быстрое наведение порядка в документах, отчетности, кадрах и торговле.
4. **База знаний / изменения рынка:** показываем, что Элдокс следит за цифровизацией шире Saby.
5. **Витрина услуг:** Saby, сайты, CRM, боты, интеграции, сопровождение.
6. **Калькулятор:** общий ориентир бюджета по направлениям.
7. **Процесс:** от задачи до работающего сценария.
8. **Доказательства / мини-кейсы:** проверяемый результат, а не обещание работ.
9. **FAQ:** снять страхи по цене, срокам, Saby, сопровождению.
10. **Форма:** оставить задачу для расчета и маршрута.

## Текстовые правила перезаполнения

Запрещенные публичные паттерны:
- “пользователь должен...”;
- “внутренний контроль...”;
- “проверяем по тройственной приемке...”;
- “технический блок...”;
- “рыба / заглушка / типовой текст”;
- “мы сделали блок, чтобы...”.

Заменять на:
- что получает клиент;
- какой риск закрывается;
- какой процесс становится понятнее;
- как проверить результат;
- какой следующий шаг сделать.

## Визуальные правила перезаполнения

1. В равных карточных группах нет случайного темного primary-блока.
2. Темный блок используется только для смены ритма, доказательства или главного вывода.
3. Одинаковые карточки не идут подряд, если они несут разные смыслы.
4. Обложки статей не повторяются внутри одной витрины.
5. У каждого CTA есть контекст: почему нажать именно сейчас.
6. Desktop проверяется отдельно от mobile: блок, который нормально смотрится на телефоне, может быть слабым на широком экране.

## Порядок внедрения

### Шаг 1. P0-тексты

Файлы:
- `v2/index.html`
- `v2/saby/index.html`
- `v2/sites/index.html`
- `v2/crm/index.html`
- `v2/knowledge/index.html`

Действия:
- убрать служебный язык;
- переписать proof-блоки;
- усилить CTA.

### Шаг 2. P0-визуальная логика

Файлы:
- `v2/styles.css`
- страницы с `pain-feature`, `solution-primary`, `proof-band`.

Действия:
- убрать необоснованные выделения;
- привести сетки карточек к цельному ритму;
- оставить акценты только там, где они функционально подписаны.

### Шаг 3. База знаний и статьи

Файлы:
- `v2/content/articles.json`
- `v2/knowledge/index.html`
- страницы статей.

Действия:
- разные изображения для первых материалов;
- усилить teaser карточек;
- добавить в статьи понятные выводы и CTA.

### Шаг 4. Скриншотная приемка

Проверить:
- главная: hero, боли, Saby, база знаний, услуги, калькулятор, процесс, proof, FAQ, форма;
- Saby: hero, проблемы, направления, proof, калькулятор, процесс;
- Sites: hero, сравнение, процесс, proof;
- CRM: hero, сравнение, процесс, proof;
- база знаний: hero, рубрикатор, карточки, proof;
- статьи: первый экран, тело, вывод, CTA;
- mobile для главной и базы знаний.

### Шаг 5. Финальный контроль

Для каждого спорного блока ответить письменно:
- почему этот визуальный акцент существует;
- какой пользовательский вопрос закрывает блок;
- какой следующий шаг становится понятнее;
- что увидит заказчик как ценность.

## Критерий готовности следующей версии

Следующая версия может быть показана как прошедшая внутренний контроль только если:

1. В публичном интерфейсе нет служебного языка фабрики.
2. Не осталось необъясненных визуальных выделений.
3. Главная читается как цельный маршрут: проблема → Saby → цифровизация → расчет → доказательство → заявка.
4. База знаний выглядит как полезная редакционная витрина, а не как шаблон с тремя одинаковыми карточками.
5. Мини-кейсы и proof-блоки дают проверяемый результат.
6. Desktop и mobile просмотрены глазами конечного пользователя.
