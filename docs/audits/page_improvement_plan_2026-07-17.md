## 1. Рамка приемки следующей итерации

Следующая сборка не должна оцениваться по принципу "страница существует". Приемка идет по четырем слоям:

1. **Структура и драматургия**
   - страница отвечает на один главный пользовательский вопрос;
   - блоки идут в логике: узнавание задачи -> ценность -> состав решения -> доказательство -> расчет/заявка;
   - нет смыслового дублирования между соседними блоками;
   - каждый блок продвигает пользователя к следующему действию.

2. **Текст**
   - текст написан для бухгалтера, руководителя или сотрудника малого бизнеса, а не для внутренней команды;
   - нет служебных формулировок вроде "состав", "рабочий порядок", "сценарий" там, где можно сказать проще;
   - каждый абзац отвечает на вопрос пользователя: "что это", "зачем мне", "как это поможет", "что делать дальше";
   - на каждой странице есть один четкий главный CTA и один мягкий CTA.

3. **Визуал**
   - блоки отличаются по роли: hero, выбор, объяснение, доказательство, расчет, FAQ, заявка;
   - нет однотипных подряд идущих карточных сеток;
   - цветовое выделение используется только если оно несет смысл;
   - на desktop страница не выглядит как растянутый mobile-макет;
   - на mobile сохраняется читаемость и понятная последовательность.

4. **Тройственная приемка**
   - разработчик: "структура реализуема, поддерживаема, без хрупких костылей";
   - заказчик: "это выглядит достойно и понятно продает услугу";
   - конечный пользователь: "я понял, что это про меня, и понимаю, куда нажать".

## 2. Системные проблемы текущей версии

### 2.1. Тексты все еще слишком рационально-служебные

Текущие страницы стали понятнее, но язык местами остается похожим на внутреннюю матрицу:

- "состав Saby";
- "задача созрела";
- "переведем ситуацию в расчет";
- "рабочий порядок";
- "запуск";
- "настройка ролей";
- "права доступа".

Эти слова не запрещены, но каждое использование нужно проверить: помогает ли оно пользователю или звучит как производственный комментарий команды.

### 2.2. Недостаточно конкретных пользовательских примеров

На страницах есть общие боли, но мало живых деталей:

- бухгалтер ищет УПД в почте;
- директор спрашивает, подписан ли акт;
- сотрудник не знает, куда отправить заявление;
- точка продаж пробила товар, а склад не видит остаток;
- отчет отправили, но статус не проверили.

Следующая итерация должна добавлять такие микро-ситуации.

### 2.3. Визуальная система пока слишком однообразная

Сейчас много блоков построено на сетках карточек. Это лучше, чем "голый текст", но недостаточно для продающего сайта. Нужны разные типы блоков:

- диагностическая развилка;
- сравнительная схема "как сейчас / как будет";
- мини-кейс;
- чек-лист;
- блок с живым примером;
- акцентная CTA-плашка;
- FAQ-аккордеон;
- блок "что нужно подготовить для расчета".

### 2.4. Не хватает доказательности

Кейсы и отзывы пока можно маркировать как заготовки, но на каждой продуктовой странице нужен хотя бы один блок:

- "пример задачи";
- "как выглядело до";
- "что настроили";
- "какой результат получил клиент";
- "что важно уточнить перед расчетом".

### 2.5. SEO-потенциал раскрыт частично

Страницы имеют заголовки и описания, но не хватает:

- H2 под конкретные поисковые намерения;
- естественного употребления запросов: "подключить Saby ЭДО", "настроить отчетность Saby", "Saby для кадров", "маркировка Saby";
- блоков "стоимость", "что входит", "как подключить", "частые вопросы";
- внутренней перелинковки между связанными страницами.

## 3. Общие правила пересборки всех страниц

### 3.1. Единая структура продуктовой страницы

Для шести страниц Saby используем одну логику, но разный визуальный ритм:

1. Hero: конкретная задача, а не абстрактный продукт.
2. "Вы узнаете себя, если...": 3-4 живых ситуации.
3. "Что дает Saby": не функции, а изменения в работе.
4. "Что подключаем": состав работ простым языком.
5. "Пример внедрения": мини-кейс или сценарий.
6. "От чего зависит цена": без универсального прайса, но с понятной логикой.
7. "Как проходит запуск": 3-5 шагов, без перегруза.
8. FAQ: 4-6 вопросов.
9. CTA: заявка на расчет.

### 3.2. Визуальный ритм

Нельзя ставить подряд:

- две одинаковые сетки карточек;
- два текстовых split-блока;
- два темных блока;
- два блока с одинаковыми маленькими карточками.

Рекомендуемый ритм:

- hero с большим визуальным акцентом;
- светлый блок с ситуациями;
- контрастный before/after;
- спокойный блок "что входит";
- доказательный мини-кейс;
- темный или акцентный блок стоимости;
- FAQ;
- форма/CTA.

### 3.3. Текстовый стандарт

Каждый блок должен пройти вопрос:

"Если это читает бухгалтер, руководитель или менеджер без технической подготовки, он понимает, зачем этот блок на странице?"

Если нет, блок переписать.

Запрещенный тип подачи:

- "Мы фиксируем задачу";
- "Выстраиваем процесс";
- "Формируем контур";
- "Прорабатываем сценарий";
- "Обеспечиваем управляемость".

Допустимый пользовательский перевод:

- "Показываем, где документ сейчас и кто должен сделать следующий шаг";
- "Настраиваем так, чтобы сотрудник видел свою задачу";
- "Собираем отправку отчетов, требования и уведомления в одном месте";
- "Помогаем выбрать только то, что нужно сейчас".

## 4. Постраничный план

## 4.1. Главная страница

### Текущая проблема

Главная перегружена двумя сильными смыслами:

- Элдокс как партнер по Saby;
- Элдокс как поставщик цифровых решений: сайты, CRM, боты.

Из-за этого страница пытается одновременно продавать Saby и расширенную цифровизацию. Для простого бухгалтера это может быть слишком сложно.

### Целевая роль страницы

Главная должна быть входной витриной:

- главный фокус: "Помогаем бизнесу подобрать, запустить и сопровождать Saby";
- второй смысл: "Если нужна не только Saby-задача, Элдокс вместе с Purrfect Apps может помочь с сайтами, CRM и ботами";
- не перегружать главную подробностями, а вести в разделы.

### Новая структура

1. Hero
   - Заголовок: "Запустим Saby под вашу задачу: документы, отчетность, кадры, торговлю или маркировку"
   - Подзаголовок: "Поможем понять, что именно нужно подключить, сколько это будет стоить и как запустить без лишней путаницы для команды."
   - CTA 1: "Подобрать Saby"
   - CTA 2: "Посмотреть направления"
   - Визуал: не абстрактный digital-арт, а деловая сцена/интерфейсная метафора документов, отчетов и статусов.

2. Блок "С чем обычно приходят"
   - 4 ситуации:
     - "Контрагенты просят электронные документы";
     - "Отчетность держится на одном человеке";
     - "Кадровые документы копятся в бумаге";
     - "Склад, касса и документы не сходятся".
   - Формат: не равные скучные карточки, а 2+2 grid с разной плотностью: первая карточка крупнее, остальные компактнее.

3. Блок "Направления Saby"
   - Короткая витрина 6 направлений.
   - Каждая карточка: название, для кого, что решает, ссылка.
   - CTA: "Открыть каталог Saby".

4. Блок калькулятора-заглушки
   - Расположить после направления Saby.
   - Логика: пользователь уже частично сформировал потребность.
   - Текст: "Получите ориентир перед разговором: выберите направление, регион и масштаб задачи."
   - Вкладки: Saby / сайты и боты / CRM.

5. Блок "Не только Saby"
   - Мягкий переход к сайтам, CRM и ботам.
   - Не делать равным главному смыслу.
   - Формулировка: "Когда Saby закрывает учет и документы, следующим шагом часто становятся заявки, сайт, CRM и автоматизация общения с клиентами."
   - CTA: "Посмотреть цифровые решения".

6. Блок доказательности
   - Пока можно использовать 2-3 смоделированных кейса с пометкой для внутренней замены.
   - Формат: "задача -> что сделали -> результат".

7. База знаний / новости
   - На главной только 3 карточки.
   - Роль: показать экспертность и регулярное обновление.
   - Кнопка: "Все публикации".

8. Финальный CTA
   - "Опишите задачу — подскажем, с чего начать"
   - Форма остается общей.

### Текстовые улучшения

Убрать или переписать:

- "цифровизация бизнеса" как главный термин;
- "процессы", "контур", "сценарии";
- длинные объяснения партнерства.

Добавить:

- "не знаете, какой Saby нужен";
- "поможем не купить лишнее";
- "сначала считаем ориентир, потом уточняем смету";
- "объясняем команде, как пользоваться после запуска".

### Визуальные улучшения

- Уменьшить количество карточных сеток.
- Добавить один сильный горизонтальный блок с контрастом.
- Разнести CTA так, чтобы они не были случайными.
- Убрать визуальные стыки, где цветной блок прилипает к следующему.

### SEO

Главные запросы:

- "подключить Saby";
- "настройка Saby";
- "Saby для бизнеса";
- "внедрение Saby";
- "Saby ЭДО отчетность кадры торговля".

Нужны H2:

- "Какие задачи Saby помогает закрыть";
- "От чего зависит стоимость Saby";
- "Как проходит запуск Saby";
- "Когда бизнесу нужны сайты, CRM и боты".

## 4.2. Каталог "Направления Saby"

### Текущая проблема

Каталог стал понятнее, но все еще выглядит как промежуточная страница, а не как полноценный навигационный центр. Hero нормальный, но ниже нужны более сильная навигация и пользовательская развилка.

### Целевая роль

Каталог должен отвечать на вопрос:

"Я не знаю, какой Saby мне нужен. С чего начать?"

### Новая структура

1. Hero
   - Заголовок: "Выберите не лицензию, а задачу, которую нужно решить"
   - Подзаголовок: "Если вы впервые разбираетесь в Saby, начните с того, что сейчас мешает работе: документы, отчеты, кадры, склад, торговля или маркировка."

2. Быстрая развилка
   - Формат: 6 интерактивных/визуально кликабельных плиток.
   - Каждая плитка:
     - "Если контрагент просит электронные документы -> ЭДО";
     - "Если отчетность держится на памяти -> Отчетность";
     - "Если бухгалтерия собирает первичку вручную -> Бухгалтерия";
     - "Если сотрудники подписывают документы на бумаге -> Кадры";
     - "Если склад и касса не сходятся -> Торговля";
     - "Если товар под маркировкой -> Маркировка".

3. Сравнительный блок
   - "Что выбрать: ЭДО, отчетность, учет, кадры, торговля, маркировка"
   - Формат: таблица или compact cards.
   - Колонки: задача / кому подходит / что будет после запуска / перейти.

4. Блок стоимости
   - "Почему цена зависит от региона и состава задачи"
   - Дать честное объяснение без оправданий.

5. CTA
   - "Не уверены, что выбрать? Опишите ситуацию."

### Текстовые улучшения

Заменить:

- "Каталог Saby" как сухой заголовок;
- "сначала простая логика" на более живое: "Начните с того, что болит в работе".

Добавить:

- примеры "если у вас...";
- короткие объяснения для неспециалиста;
- переходы к детальным страницам как естественный следующий шаг.

### Визуальные улучшения

- Сделать все карточки направлений кликабельными и визуально равноправными.
- Выделение цветом использовать для hover/active, а не для случайной первой карточки.
- Добавить иконки/мини-иллюстрации по направлениям.
- Уменьшить высоту длинных карточек; сейчас длинный текст делает каталог тяжелым.

### SEO

Нужны естественные фразы:

- "какой Saby выбрать";
- "направления Saby";
- "Saby ЭДО";
- "Saby отчетность";
- "Saby кадры";
- "Saby торговля";
- "Saby маркировка".

## 4.3. ЭДО

### Текущая проблема

Страница стала понятнее, но hero слишком длинный, а блоки ниже остаются похожими на шаблон. Не хватает ощущения, что это страница именно про ЭДО, а не универсальная заготовка.

### Целевая роль

Объяснить простому клиенту:

- что такое ЭДО в Saby;
- когда без него уже неудобно;
- что нужно для запуска;
- как не потерять документы после подключения.

### Новая структура

1. Hero
   - Заголовок: "Подключим ЭДО в Saby, чтобы документы не терялись в почте и бумаге"
   - Подзаголовок: "Настроим обмен с контрагентами, подписи, статусы и понятный порядок работы для бухгалтерии."
   - CTA: "Рассчитать ЭДО"

2. Блок "Когда пора подключать ЭДО"
   - 4 ситуации:
     - контрагент просит УПД через оператора;
     - документы ищут в почте;
     - директор не видит, подписан ли акт;
     - часть документов электронная, часть бумажная.

3. Блок "Что изменится"
   - До: документы в письмах, сканах, папках.
   - После: отправка, подпись, статус и хранение в Saby.
   - Визуал: горизонтальная схема.

4. Блок "Что подключаем"
   - обмен с контрагентами;
   - пользователи и права подписи;
   - шаблонный порядок работы;
   - проверка первого обмена.

5. Мини-кейс
   - "Компания получала УПД от 12 контрагентов по почте..."
   - "Настроили обмен и роли подписи..."
   - "Бухгалтерия перестала искать финальные версии в письмах."

6. Цена
   - регион;
   - количество организаций;
   - пользователи;
   - объем документов;
   - сопровождение.

7. FAQ
   - "Можно начать с одного контрагента?";
   - "Что делать с бумажными документами?";
   - "Нужна ли электронная подпись?";
   - "Сколько занимает запуск?";

### Текстовые улучшения

Убрать:

- "электронный обмен документами становится задачей на ближайший запуск" как тяжелую формулировку.

Заменить на:

- "Если контрагент уже просит ЭДО, лучше не откладывать: чем раньше настроить порядок, тем меньше документов потеряется в переходный период."

### Визуальные улучшения

- Добавить визуальную метафору документа: входящий УПД -> подпись -> статус -> архив.
- Блок "До/После" сделать главным визуальным элементом, а не двумя карточками.
- FAQ оформить аккордеоном с явной кликабельностью.

### SEO

Запросы:

- "подключить Saby ЭДО";
- "настроить ЭДО Saby";

- "Saby ЭДО для контрагентов".

## 4.4. Отчетность

### Текущая проблема

Страница говорит о проблеме, но недостаточно конкретно показывает, что именно получает бухгалтер или руководитель.

### Целевая роль

Показать, что Saby помогает:

- сдавать отчеты;
- отслеживать требования;
- видеть статусы;
- меньше держать сроки вручную.

### Новая структура

1. Hero
   - Заголовок: "Настроим отчетность Saby, чтобы сроки, требования и статусы были под контролем"
   - Подзаголовок: "Подключим отправку отчетов, уведомления и порядок работы с требованиями без ручного контроля в таблицах."

2. Блок "Когда это уже нужно"
   - сроки держатся на памяти бухгалтера;
   - требования приходят в разные каналы;
   - руководитель узнает о проблеме поздно;
   - несколько организаций/филиалов.

3. Блок "Что будет видно"
   - ближайшие сроки;
   - что отправлено;
   - что принято;
   - где требуется ответ.

4. Блок "Что подключаем"
   - организации;
   - пользователи;
   - отчетность;
   - требования;
   - уведомления.

5. Мини-кейс
   - "В компании отчеты контролировал один бухгалтер..."
   - "Подключили уведомления и порядок проверки..."
   - "Руководитель стал видеть риски заранее."

6. Цена
   - количество организаций;
   - регион;
   - набор отчетности;
   - пользователи;
   - сопровождение.

7. FAQ
   - "Можно подключить только отчетность?";
   - "Можно ли работать с несколькими организациями?";
   - "Что с требованиями?";
   - "Как понять точную цену?";

   - где сейчас теряются заявки;

4. Поля для CRM:
   - сколько менеджеров;
   - где сейчас заявки;
   - что нужно видеть руководителю.

5. После отправки
   - не просто "отправлено";
   - показать: "Мы получили задачу. Обычно для ориентира по Saby нужно уточнить регион, продукт и пользователей. Мы вернемся с вопросами или расчетом."

- Форма должна быть не техническим блоком, а спокойным завершающим экраном.
- Добавить подсказки в полях.

## 6. Чек-лист тройственной приемки следующей версии

Перед показом владельцу каждая страница проверяется:

### Разработчик

- нет битых ссылок;
- нет переполнений на desktop/mobile;
- меню работает на всех страницах;
- структура поддерживаемая;
- нет локальных артефактов только в Codex workspace.

### Заказчик

- страница выглядит достойно;
- понятно, что продает Элдокс;
- нет ощущения "рыбы";
- страница не стыдная для клиента;
- CTA расположен логично.

### Конечный пользователь

- за первые 5-7 секунд понятно, куда он попал;
- он узнает свою проблему;
- он понимает, что будет после заявки;
- у него есть желание нажать CTA;
- язык не требует технической подготовки.

## 7. Контрольная формула для сборки

Для каждого блока перед реализацией заполнить:

`страница -> блок -> задача блока -> пользовательский вопрос -> текст -> визуальный тип -> CTA/следующий шаг -> критерий приемки`.

Если у блока нет ответа на пользовательский вопрос, блок не собирается.
