# Прототипы визуальных решений для пересборки

## Цель визуальной системы

Сайт должен выглядеть как спокойная B2B-витрина технологического партнера, а не как набор одинаковых карточек. Визуальная задача: показать порядок, компетентность и понятный путь выбора Saby.

## Общие принципы

- Не больше двух подряд секций с одинаковой сеткой карточек.
- Hero не должен занимать огромный экран без необходимости.
- Каждый продуктовый блок должен иметь свой визуальный тип: схема, сравнение, карточки ситуаций, чек-лист, таймлайн, FAQ.
- Желтая кнопка используется только для главного CTA.
- Темные блоки использовать как смысловые акценты, а не просто для красоты.
- На desktop страница должна быть плотной и уверенной; не растягивать мобильные решения на широкий экран.

## Прототип 1. Hero продуктовой страницы

Назначение: быстро объяснить, кому нужна страница.

Структура:

- слева: kicker, H1, короткий lead, две кнопки;
- справа: "Что обычно уточняем" или мини-схема;
- низ hero не должен требовать скролла, чтобы понять тему.

Пример для ЭДО:

```text
Kicker: Saby ЭДО
H1: Электронные документы с контрагентами без поиска оригиналов в почте и папках
Lead: Подходит, если контрагенты просят ЭДО, а акты, УПД и счета пока согласуются в письмах, чатах и бумажных папках.
CTA 1: Рассчитать ЭДО
CTA 2: Посмотреть, что входит
Right: Документы / роли / подписи / статусы / обучение
```

## Прототип 2. Блок "Когда вам сюда"

Назначение: пользователь должен узнать свою ситуацию.

Визуал:

- 4 карточки ситуаций;
- каждая карточка не равна другой по высоте механически, но сетка аккуратная;
- первая карточка может быть выделена только если она действительно "самый частый вход".

Содержание карточки:

```text
Заголовок: Контрагент просит ЭДО
Текст: Нужно отправлять УПД, акты и счета электронно, но внутри компании еще нет понятного порядка подписи и контроля статусов.
Мини-метка: частый старт
```

## Прототип 3. Блок "Что меняется после запуска"

Назначение: показать пользу, а не состав лицензии.

Визуал:

- сравнение "до / после";
- две колонки;
- слева спокойный светлый блок "сейчас";
- справа акцентный блок "после запуска".

Пример:

```text
Сейчас:
- документы ищут в почте;
- статус уточняют вручную;
- оригиналы хранятся в папках;
- ответственный вспоминается по переписке.

После запуска:
- видно, что отправлено и подписано;
- документы хранятся в электронном архиве;
- роли и действия понятны;
- команда знает, куда заходить.
```

## Прототип 4. Схема работы

Назначение: дать ощущение управляемого маршрута без слова "маршрут", если оно звучит технически.

Визуал:

- горизонтальная линия на desktop;
- вертикальная на mobile;
- 4-5 шагов с короткими подписями.

Пример для отчетности:

```text
1. Выбираем организации и регион.
2. Смотрим, какие отчеты и требования нужны.
3. Настраиваем доступы и уведомления.
4. Показываем ответственным, где смотреть сроки и статусы.
5. Помогаем после первого отчетного периода.
```

## Прототип 5. "Что входит в запуск"

Назначение: показать работу Элдокс, а не просто список возможностей Saby.

Визуал:

- карточки с иконками или номерами;
- каждая карточка начинается с действия Элдокс.

Формат:

```text
Подбираем состав
Смотрим задачу, регион, пользователей и текущие сервисы, чтобы не покупать лишнее.

Настраиваем роли
Помогаем определить, кто отправляет, подписывает, проверяет и контролирует.

Обучаем команду
Показываем сотрудникам, где выполнять ежедневные действия.
```

## Прототип 6. "Что влияет на цену"

Назначение: честно объяснить расчет, не пряча цену.

Визуал:

- темный или контрастный блок;
- 5 факторов в строку/сетку;
- рядом CTA "Получить расчет".

Факторы:

- регион;
- что именно нужно запустить в Saby;
- количество пользователей;
- количество организаций/точек/товаров;
- настройка, обучение, сопровождение.

## Прототип 7. "Ошибки при самостоятельном выборе"

Назначение: показать экспертность и причину обратиться.

Визуал:

- список из 3 ошибок;
- каждая ошибка имеет "как исправляем".

Пример:

```text
Ошибка: купить слишком широкий состав.
Как исправляем: сначала отделяем обязательное от того, что можно подключить позже.
```

## Прототип 8. FAQ

Назначение: снять возражения перед заявкой.

Визуал:

- accordion;
- 5-7 вопросов;
- не прятать критически важные вопросы слишком низко.

Базовые вопросы:

- Можно ли начать с одного направления?
- Почему цена зависит от региона?
- Что делать, если Saby уже есть, но им неудобно пользоваться?
- Можно ли подключить только ЭДО/отчетность/кадры?
- Что нужно подготовить для расчета?

## Прототип 9. Заявка

Назначение: не просто форма, а следующий понятный шаг.

Визуал:

- слева: что произойдет после отправки;
- справа: форма;
- поле "Задача" обязательно;
- текст после отправки без технических слов.

Текст:

```text
После заявки мы уточним регион, что именно нужно запустить в Saby, пользователей и текущий порядок работы. Затем подготовим понятный ориентир по составу запуска и стоимости.
```

## Прототип 10. Карточка направления в каталоге

Назначение: выбрать страницу.

Состав:

- название;
- "когда вам сюда";
- 3 коротких пункта;
- CTA.

Пример:

```text
ЭДО
Когда контрагенты просят электронные документы, а компания пока держит акты, УПД и счета в почте и папках.

Подходит, если нужно:
- обмениваться документами;
- видеть статус подписи;
- объяснить сотрудникам новый порядок.

CTA: Разобраться с ЭДО
```

## Запреты на следующую сборку

- Не делать подряд три блока одинаковых карточек.
- Не ставить форму сразу после одной общей секции.
- Не выделять карточку цветом без смысловой причины.
- Не использовать "технические" подписи типа "контентный контур", "диагностика", "процесс" без проверки уместности.
- Не делать страницы направлений слишком короткими: они должны быть SEO-полезными и продающими.
