# Контрольный лист пересборки и тройственной приемки

## Перед сборкой страницы

Страницу нельзя трогать в HTML/CSS, пока не заполнены:

- аудитория;
- ситуация пользователя;
- главный смысл страницы;
- список блоков;
- визуальный тип каждого блока;
- черновой текст каждого блока;
- CTA;
- критерий приемки.

## Проверка структуры

- На странице есть один главный смысл.
- Страница не пытается продать сразу все направления.
- Блоки идут в причинно-следственной логике.
- Форма появляется после того, как пользователь понял пользу и состав работы.
- Нет смысловых дублей.
- Нет блока, который существует только потому, что "так принято".

## Проверка визуала арт-директором

- Две подряд секции не выглядят одинаково.
- Цветовое выделение имеет смысловую причину.
- Активное состояние меню показывается только где нужно.
- Desktop-версия не выглядит как растянутая мобильная.
- Нет больших пустот без смысловой паузы.
- CTA заметен, но не кричит.
- Карточки имеют разный смысловой вес, а не просто одинаковый шаблон.

## Проверка текста

- Первый экран отвечает на вопрос "куда я попал и зачем мне это".
- В тексте есть конкретные рабочие ситуации.
- Термины Saby объяснены простым языком.
- Нет внутренних слов без пользы для пользователя.
- "Процесс", "сценарий", "контур", "диагностика", "цифровизация" проверены по месту, а не запрещены механически.
- CTA связан с задачей страницы.
- Текст можно прочитать бухгалтеру/руководителю без дополнительного перевода.

## Проверка конечным пользователем

Пользователь должен ответить "да" на вопросы:

- Я понимаю, какую задачу решает эта страница.
- Я понимаю, относится ли это ко мне.
- Я понимаю, что входит в помощь Элдокс.
- Я понимаю, почему цена зависит от параметров.
- Я понимаю, что написать в форме.
- У меня стало больше уверенности оставить заявку.

## Проверка заказчиком

Заказчик должен ответить "да" на вопросы:

- Страница выглядит профессионально.
- Страница не стыдная для показа клиенту.
- Тексты не похожи на техническую заготовку.
- Элдокс выглядит компетентно.
- Saby-first позиционирование не потеряно.
- Purrfect Apps не перетягивает фокус, если речь о Saby-странице.

## Проверка разработчиком

- Header единый и предсказуемый.
- Dropdown не исчезает при наведении.
- Активные состояния не вводят в заблуждение.
- HTML читаемый, не превращен в однострочную кашу.
- Страницы открываются локально и публично.
- Нет битых ссылок.
- Нет технических сообщений в UI.
- Sitemap обновлен, если менялись URL.

## Минимальная приемка P0-страниц

Каждая из страниц должна иметь:

- hero;
- блок "когда вам сюда";
- блок "что это дает";
- блок "что входит";
- блок "что влияет на цену";
- блок "как работает Элдокс";
- FAQ;
- заявка.

Допускается менять порядок, если это улучшает драматургию, но нельзя терять смысл.

## Запрет на приемку

Страницу нельзя признавать готовой, если:

- ее можно описать как "заготовка";
- текст можно без потери смысла перенести на любую другую страницу;
- пользователь не понимает, какую конкретную задачу решает продукт;
- форма стоит раньше, чем построено доверие;
- визуал держится только на одинаковых карточках;
- есть активные состояния меню без причины;
- отдел контроля проверил только наличие блоков, а не качество блоков.

## Итерационный процесс

1. Собрать страницу по матрице.
2. Провести внутреннюю проверку разработчика.
3. Провести арт-директорскую проверку.
4. Провести текстовую проверку.
5. Провести пользовательскую проверку.
6. Исправить.
7. Повторять до прохождения без критических замечаний.
8. Только после этого показывать владельцу.

## Что логировать

- версия страницы;
- какие блоки изменены;
- какие замечания закрыты;
- какие замечания остались;
- что показала тройственная приемка;
- путь к каноническому артефакту на H:;
- commit в GitHub, если изменения значимые.
